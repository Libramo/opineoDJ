# AI Workflow Rules — OpineoDJ

## Environment

This project is built using **Claude Code inside VS Code**.
Claude Code reads all context files directly from the repository.
Always read the relevant context files at the start of each session
before writing any code.

### Session startup checklist
1. Read `progress-tracker.md` → understand current phase and goal
2. Read `architecture.md` → confirm stack and invariants
3. Read `code-standards.md` → confirm conventions before touching any file
4. Read `ui-context.md` → before touching any component or style
5. Check `npm run build` passes before starting new work

---

## Approach

Build OpineoDJ incrementally using a spec-driven workflow. The context files
define what to build, how to build it, and the current state of the project.
Always implement against these specs — do not infer or invent behavior not
defined here.

---

## Claude Code specific rules

- Always operate from the **project root** — never cd into subdirectories
  and leave files there
- After every file creation or edit, confirm the file exists and is valid:
  `cat` the file or run `tsc --noEmit` for TypeScript files
- Use `npm run build` to verify the full project compiles — do not skip this
- Never run `drizzle-kit push` or `drizzle-kit migrate` without explicit
  instruction — schema changes to the database are irreversible
- When installing packages, always use `npm install` — never `yarn` or `pnpm`
  unless the project already uses them
- Do not create files outside the defined folder structure in `code-standards.md`
- If a terminal command fails, read the full error before retrying —
  do not blindly retry the same command

## Recommended Build Order

```
Phase 1 — Foundation
  1.1  Full Supabase schema (SQL + Drizzle migrations)        ✅ COMPLETE
  1.2  Next.js project initialized (App Router, Tailwind v4, shadcn/ui, better-auth)
  1.3  Base design system (CSS tokens, shadcn components installed)

Phase 2 — Collection
  2.1  Admin dashboard — layout + auth (better-auth)
  2.2  Survey creation (campaign → wave → questions → quotas)
  2.3  Public collection form (`/survey/[slug]`)
  2.4  Response submission + storage
  2.5  Resend email notifications (on submission + milestones)

Phase 3 — Basic Analysis
  3.1  Simple aggregations (%, averages, distributions) — PostgreSQL RPC
  3.2  Real-time public dashboard (`/results/[slug]`) + Supabase Realtime
  3.3  Analyst dashboard — raw results view

Phase 4 — Advanced Analysis
  4.1  FastAPI microservice setup (Vercel Python runtime)
  4.2  Weighting engine (raking / iterative proportional fitting)
  4.3  Confidence intervals and margins of error
  4.4  Cross-tabulations (flat breakdown / cross-tab explorer)
  4.5  Inter-wave trend tracking (delta between waves)

Phase 5 — Export & Integration
  5.1  PDF export (react-pdf, branded report)
  5.2  Excel/CSV export (xlsx)
  5.3  Public REST API (aggregated data per survey)
  5.4  Webhooks for external pipelines
```

## Scoping Rules

- Work on **one unit at a time** — one unit = one screen, one route,
  one major component, or one endpoint
- Prefer **small, verifiable increments** over large speculative changes
- Do not combine unrelated system boundaries in a single implementation step

---

## When to Split Work

Split an implementation step if it combines:

- UI changes and backend logic changes
- Multiple unrelated API routes
- Behavior not clearly defined in the context files
- DB schema changes AND application logic changes

If a change cannot be verified end to end quickly, the scope is too broad — split it.

---

## Handling Missing Requirements

- Do not invent product behavior not defined in the context files
- If a requirement is ambiguous, resolve it in the relevant context file
  BEFORE implementing
- If a requirement is missing, add it as an open question in
  `progress-tracker.md` before continuing

---

## Protected Files

Do not modify without explicit instruction:

- `components/ui/*` — components generated by the shadcn CLI
- `lib/db/migrations/*` — migration files generated by Drizzle Kit
- `lib/db/schema.ts` — only modify when explicitly changing the data model
- `api/py/requirements.txt` — except to add a justified dependency

---

## Keeping Docs in Sync

Update the relevant context file whenever implementation changes affect:

- System architecture or boundaries
- Data model (DB schema)
- Code conventions or standards
- Feature scope

Always update `progress-tracker.md` at the end of every session.

---

## Before Moving to the Next Unit

1. The current unit works end to end within its defined scope
2. No invariant defined in `architecture.md` was violated
3. `progress-tracker.md` reflects the completed work and is up to date
4. `npm run build` passes without errors
5. TypeScript compiles without errors (`tsc --noEmit`)
